摘要：完成地图生成接口，实现简单的用户信息管理功能

1、地图生成算法

1.1 算法思路

[2024/4/29]

考虑两个地图

可以发现，左边的地图可以很快看出解，而右边的地图却很难看出入手点。A中矩阵比较聚集，矩阵平均重合面积大，矩阵边界比较明显；B中矩阵比较分散，矩阵平均重合面积较小，边界比较不明显。

由此指制定量化规则：

  • 使用 $Var(A)$ 来描述子矩阵的分散程度，参考方差的定义，取每个矩阵的中心坐标作为矩阵的位置，定义$Var(A) = \sum_{i=1}^n{d( M_i - C )^2}/(R \times n)$，其中$C = \sum_{i=1}^n w(M_i)\times C(M_i)$，$w$ 为矩阵权重，$R=max_{i=1}^n { d(M_i-C) }$

  • 定义至少三边都有空元素或边界的矩阵的边界值$E(M)$为周边空元素及边界的总和，地图的边界值为全部子矩阵的边界值求和再除以全部子矩阵的边界大小之和，即$E(A) = {\sum_{i=1}^nE(M_i)}/\sum_{i=1}^n4*width(M_i)$。

对以上两个地图，有

$Var(A) = 0.4622$ $Var(B) = 0.5762$

$E(A) = 0.375$ $E(B)=0.188$

故由此得出生成算法的思路：先生成随机地图，然后使用优化算法将地图的度量调整至指定值附近

3.2.3 头像获取功能开发

接下来头像的获取就成为了开发的第二个大难关。由于整个 core 模块将运行在子线程，而获取头像需要使用 QFileDialog 进行操作，但是子线程中不允许创建窗口，于是就必须由子线程向主线程发射信号，让主线程来操作控件。

为此，在 common 模块中封装 UiHelper，定义信号与槽函数进行通信，并在 UiHelper中创建窗口，后期如有类似需求可以在此基础上开发。

接着利用 CoreController 类，封装部分需要向外通信的函数。由于信号连接需要获取实例，故将 CoreController 改为单例模式，并更改 app.cpp 中的相应初始化逻辑。

其中 getAvatarPath 方法中利用 QEventLoop 阻塞等待主线程信号（这种处理方式也许不是最优解，但是懒得考虑太多了）。其中，UiHelper 的 onGetAvatarPath 信号会带着选择后的路径传到 CoreController，onSendAvatarPath 信号会让 getAvatarPath 方法退出事件循环，终止阻塞返回结果。

最后就是头像获取逻辑的编写。

在这部分其实有几个地方考虑得并不是那么周到。

  • 首先，是刚刚完成的头像获取逻辑。本来这种地方应该采用异步编程的方式进行开发，不应该就单纯利用事件循环来阻塞线程，但是考虑到已有的 Dispatcher - Handler 架构比较难采用异步来开发处理逻辑，而且引入异步还会造成开发复杂度大幅提升，于是就选了更加偷懒的办法。
  • 其次，是线程向外通信的架构。当初在设计程序架构的时候只考虑到了数据传输，封装了 Dispatcher 通过信号来与 QWebChannel 进行交互，没有考虑到从下到上传递信息的需求，到时临时开发 UiHelper 类，而且还把 Controller 的功能复杂化了，不符合 SRP 原则（但是我真的懒得改架构了）
  • 最后，是图片处理用了 data url。当初在设计通信管道的时候只考虑到了 Json 对象的交互，没有考虑到图片的交互，导致后期前后端通信只能基于 Json。后来本来想了补救办法，就是向前端传递图片的 file:// 链接，但是经过测试，QWebEngine 环境下无法读取本地数据，只能读取 qrc 中的数据。于是最后只能通过 data url 进行图片传输，而且为了便捷还直接把图片的 Base64 数据直接存在本地的 Json 数据文件里了（别骂了，但是我真的懒得改）

[2024/5/18] 终于完成了第一个功能的开发啊。。本来以为是很简单的事情，结果硬生生拖了一个星期，后面的开发任重道远啊（）
